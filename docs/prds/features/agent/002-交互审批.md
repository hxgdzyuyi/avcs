# 002 交互审批

状态：Draft  
领域：`lib/avcs/agent`、`lib/avcs_web/channels`  
关联 PRD：`docs/prds/features/agent/001-Codex-Agent调用.md`、`docs/prds/features/ui/003-Codex聊天区对齐.md`

## 1. 目标

让用户在 Agent 运行中对需要审批的动作（沙箱逃逸、写入受限路径、网络访问、MCP 调用等）做出批准或拒绝决策，而不是一律放行或一律拒绝。

本能力是 `ui/003` §9.2 权限方式的可选升级项：`ui/003` 默认 `Auto`（`approvalPolicy: never`，非交互）；当用户把权限方式调到需要审批的档位时，由本 PRD 定义的审批流接管。

## 2. 范围与定位

1. 仅当 `approvalPolicy` 非 `never` 时启用，例如 `untrusted`、`on-failure`、`on-request`。
2. 审批 UI 复用 `ui/003` 的 turn 渲染，审批卡片落在触发审批的 turn 内。
3. 不在 Avcs 内部实现独立的风险判定；判定由 Codex 完成，Avcs 只负责把审批请求呈现给用户并回传决策。

## 3. 协议依据

基于 Codex app-server v2 协议（schema 见 `priv/codex_app_server/schemas`）：

1. `thread/start` 的 `approvalsReviewer` 设为 `user`，把审批路由给用户。
2. 服务端通知 `item/autoApprovalReview/started`、`item/autoApprovalReview/completed` 表示审批开始与结束。
3. 客户端调用 `thread/approveGuardianDeniedAction` 批准被拒动作，参数含 `threadId` 与序列化的 `GuardianAssessmentEvent`（`event`）。

⚠️ 风险：上述审批通知在协议中标注为 `[UNSTABLE]`，「expected to change soon」。实施时数据层必须保留 Codex 原始 payload，并把通知解析隔离在单独适配层，便于协议变化后快速跟进。

## 4. 后端结构性改造

1. `codex_client.ex` 现硬编码 `approvalPolicy: "never"`、`sandboxPolicy: workspaceWrite`，且 developer instructions 写有「Do not ask follow-up questions」。启用审批需：
   - 改为按 thread/turn 选择传入 `approvalPolicy` / `sandboxPolicy` / `approvalsReviewer`。
   - 放宽「不问追问」约束，允许在需要审批的动作上交还控制权。
2. 当前 `run_turn` 是单次同步收集（`collect_turn` 阻塞 receive 循环）。交互审批要求运行中接收用户决策并回传 Codex：
   - 运行任务需支持中途接收审批决策消息（例如向运行 Task 发送消息或经独立通道注入），而非一次性同步跑完。
   - 这是 runner / codex_client 的结构性改动，是本能力的主要工作量。

## 5. WebSocket 事件

客户端发送：

1. `approval:respond`：回传审批决策（approve / deny + 关联的审批标识），后端据此调用 `thread/approveGuardianDeniedAction`。

服务端推送：

1. `approval:requested`：映射 `item/autoApprovalReview/started`，要求用户对被审动作做决策。
2. `approval:resolved`：映射 `item/autoApprovalReview/completed`，审批完成后更新审批卡片状态。

所有审批事件必须携带 `thread_id`、`turn_id`，并在存在时携带 `target_item_id` 与 `review_id`。

## 6. 前端审批卡片

1. 审批请求在对应 turn 内渲染为可操作审批卡片，展示被审动作摘要（命令、写入路径、网络访问等）。
2. 提供批准 / 拒绝操作。
3. 待审批期间 turn 状态为「等待审批」，不计为 `completed` 或 `failed`。
4. 审批卡片归属 `thread_id`、`turn_id` 和（存在时）`target_item_id`，切换 thread 不串显。
5. 审批结束后卡片更新为已批准 / 已拒绝的稳定状态，刷新页面仍可追踪。

## 7. 数据

复用 `ui/003` §10 中 Thread/Turn 的 `approval_policy`、`sandbox_mode` 字段；审批请求与结果作为 item（或 item 上的 payload）落库，保留 Codex 原始 `[UNSTABLE]` payload。

## 8. 不做事项

1. 不在 Avcs 内部重写风险判定逻辑。
2. 不在 `approvalPolicy: never`（默认 `Auto`）下渲染审批 UI。
3. 不把审批状态做成只存在内存、刷新后不可追踪的唯一记录。

## 9. 验收标准

1. 权限方式为需审批档位时，`thread/start` 携带 `approvalsReviewer: user`。
2. Agent 触发审批时，对应 turn 内显示审批卡片并展示动作摘要。
3. 用户批准 / 拒绝后正确回传 Codex，运行继续或动作被拒。
4. 待审批期间 turn 状态为等待审批，不误判为完成或失败。
5. 切换 thread 不会把另一个 thread 的审批卡片显示到当前 thread。
6. 审批结果落库，刷新页面仍可看到该审批的最终状态。
