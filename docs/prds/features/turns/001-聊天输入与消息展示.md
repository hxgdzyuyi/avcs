# 003 聊天输入与消息展示

状态：Draft  
领域：`lib/avcs/turns`

## 1. 目标

聊天区用于展示当前 thread 下的 turn/item，承载用户自然语言输入、Agent 输出、工具调用状态、错误信息和图片结果入口。

## 2. 数据模型

Turn 表示一次用户输入和 Agent 输出的完整回合。Item 表示回合中的细粒度内容。

Item 基础类型：

1. `user_message`
2. `assistant_message`
3. `tool_call`
4. `tool_result`
5. `image_asset`
6. `error`

历史编辑需要保留当前有效路径与旧路径的边界。`turns` 和 `items` 默认只展示 `invalidated_at IS NULL` 的记录；被历史编辑截断的旧 turn/item 标记 `invalidated_at` 和 `invalidated_by_item_id`，不物理删除，以便后续审计或恢复能力扩展。

## 3. 输入器

聊天输入区使用 CodeMirror 作为多行编辑器。MVP 以自然语言和 prompt 编辑体验为主，支持：

1. 多行输入。
2. 粘贴长文本。
3. 基础撤销重做。
4. 发送快捷键。
5. Agent 运行中仍可编辑下一条草稿。
6. 从剪贴板粘贴截图或图片，并自动作为当前消息图片引用上传。
7. Agent 运行中且 composer 为空时，右下角主按钮显示 Stop；composer 有文本或图片引用时，主按钮发送追加输入到当前 active turn。
8. 在调色盘按钮左侧提供书本图标 Data Provider 入口，展示 Avcs 内置 provider（如 `avcs-data-prodiver-apod`、`avcs-data-prodiver-steam`）。用户选中 provider 后立即标记为已加载，并在 composer 中回显已附加状态。

快捷键规则：

1. `Shift+Enter` 插入换行。
2. `Enter` 和 `Mod+Enter` 发送当前 prompt；发送前仍按空文本、引用上传状态和 Agent 运行状态做校验。
3. `Cmd+V` / `Ctrl+V` 粘贴剪贴板图片并上传为当前消息引用；普通文本粘贴保持 CodeMirror 默认行为。
4. 焦点在 composer 内时不触发 Board 或图层全局快捷键。

MVP 不把 CodeMirror 扩展成完整代码 IDE。

Composer 视觉要求：

1. 固定在聊天栏底部。
2. 最小高度 `148px`，最大高度 `280px`。
3. 左下角放添加图片按钮。
4. 右下角放发送按钮，使用明确的 action 色。
5. 不写长篇使用说明；placeholder 保持一句话，例如 `Describe the image you want to create...`。
6. 图片上传或粘贴上传中需要显示紧凑占位；失败时保留输入文本并显示可恢复错误。
7. 提供调色盘图标入口，用于本轮选择图片比例、图片数量和透明背景偏好。
8. 提供 Data Provider 图标入口，用于本轮选择和加载外部数据源；未选择时不改变现有发送行为。

调色盘配置：

1. 图片比例选项：`auto`、`1:1`、`16:9`、`9:16`、`4:3`、`3:4`、`3:1`、`1:3`。
2. 图片数量选项：`1` 到 `4`。
3. 透明背景为布尔开关。
4. 面板初始值来自全局软件设置 `image.default_ratio`、`image.default_count` 和 `image.transparent_background`。
5. 发送时前端把非默认配置拼接到 prompt 后面，不新增 WebSocket 字段。
6. 配置只影响本次按 Enter、`Mod-Enter` 或发送按钮发出的提示词。
7. 发送成功后恢复为全局软件设置中的图片默认值。

## 4. 消息列表

消息区以 turn 为节奏展示：

1. 用户消息右侧气泡，浅灰背景，最大宽度约 `72%`。
2. Assistant 消息左侧正文，不强制气泡化，保持长文本可读。
3. 工具调用使用紧凑状态行，例如 `image_gen running`、`bash completed`、`grep failed`。
4. 错误信息使用浅红背景或红色边线，必须说明失败来源：Agent、文件 API、WebSocket 或预览读取。
5. 图片生成结果渲染为一组可点击的资产行，而不是只显示在文本里。

资产行示例：

```text
[image icon] coffee-logo-v03.png                         [locate icon]
[image icon] coffee-logo-v04.png                         [locate icon]
[image icon] coffee-logo-v05.png                         [locate icon]
```

点击资产行应在画板中定位并选中对应对象；如果画板在当前窗口隐藏，则将图片加入聊天引用或打开预览。

用户消息编辑：

1. MVP 只允许编辑非 `steered` 的起始 `user_message`。
2. Agent 正在运行时禁用历史消息编辑；用户需要先 Stop 或等待当前 turn 收敛。
3. 保存历史用户消息会更新该 item 文本和对应 `turns.user_text`，并让该消息之后的本地 turn/item 退出当前有效路径。
4. 同一 turn 中位于目标 user item 之后的 assistant/tool/image/error item 也退出当前有效路径。
5. 图片引用、mask edit 和 data provider 仍沿用原 user item payload，MVP 不在历史编辑中增删引用。

## 5. 发送流程

1. 用户在 CodeMirror 中输入文本。
2. 如果本轮调色盘配置存在非默认项，前端把配置拼接到文本后面，例如 `Image settings: aspect ratio 16:9; image count 2; transparent background.`。
3. 前端把最终文本、当前引用资产列表和可选 `data_provider` 通过 WebSocket 发送给 Phoenix；如有粘贴图片仍在上传，发送需等待上传完成。
4. 如果处于新会话准备状态，系统先创建 thread；普通续聊则使用当前 thread。
5. Agent 运行参数优先使用本次发送显式传入的 model / effort / approval policy / sandbox mode；未传入时回退当前 thread 默认值，再回退全局软件设置 `agent.*`，最后使用 Avcs 内置默认值。Codex-only effort / approval / sandbox 只对 Codex Harness 生效；AvcsAgent 使用 `agent.avcs_agent.*` 设置。
6. 如果 `data_provider` 存在，后端校验 provider 已加载且 slug 受支持，并将 provider 元信息持久化到 turn 配置和 user item payload。
7. 如果目标 thread 正在运行且本轮输入非空，后端优先调用当前 Agent Harness 的 steer 能力。Codex Harness 通过 `turn/steer` 发送到 active turn；AvcsAgent 将追加输入放入 pending steer 队列，并在模型响应或工具批次之间的安全点作为 turn continuation 注入当前 turn，不创建新 turn。
8. 如果目标 thread 未运行，后端为目标 thread 创建 turn 和 `user_message` item。
9. 后端经 Agent Harness 调用当前 runtime 发起 turn。Codex Harness 使用 `turn/start`；AvcsAgent 使用 OpenAI-compatible streaming loop。如果本轮携带 `data_provider`，AvcsAgent 在 context 中注入 provider skill，引导模型先调用受控 `bash` data provider descriptor；`bash` 由 Phoenix 运行内置脚本，把来源图片写入 `work/` 并入库为 provider asset，随后模型可调用 `image_gen` 生成 `output/` 图片。
10. Agent 输出、工具调用和错误被持续解析为 item 并推送给前端。
11. 当前 turn 完成后，状态更新为 `completed`、`failed` 或 `interrupted`。
12. 开发/验收场景可使用 `mix mock_many_turns <project_sqlite_path> <turn_count>` 直接生成离线 `thread/turn/item`，用于本地 chat pane 验证。

Agent Harness 是后端内部运行时边界，基础版支持全局 `codex | auto | avcs_agent`，默认 `codex`。前端 `message:send`、`turn:stop` 和 `message:edit_rerun` payload 不携带 harness 标识，也不出现 harness-specific 分支；实际 harness 写入 thread/turn 的 `agent_harness`。

AvcsAgent 的模型可见工具名采用 pi-agent 风格（`read`、`write`、`edit`、`bash`、`grep`、`find`、`ls`），但它们全部是 Avcs-native 后端受控工具，不是开放 shell 或开放文件系统。默认 active tools 包含 `image_gen`、`read`、`ls`、`find`、`grep`、`bash`；`write`、`edit` 只有显式开启时可用，且仍受项目路径、`.avcs/`、SQLite、secret-like 文件和 symlink escape 检查约束。

历史编辑重跑流程：

1. 前端通过 `message:edit_rerun` 提交目标 `item_id` 和新文本。
2. 后端在单个项目 SQLite transaction 中校验、更新锚点用户消息、失效后续本地路径，并把锚点 turn 重置为 `queued` 或 `in_progress`。
3. 如果 thread 已有关联 `remote_thread_id`，runner 请求当前 Agent Harness 准备重跑。Codex Harness 优先 fork 后 rollback，再从编辑后的消息重新 `turn/start`；AvcsAgent 记录本地 `thread/fork` / `thread/rollback` trace，并基于 Avcs SQLite 当前有效路径重新构建上下文，不调用或复刻 Codex app-server 协议。
4. Codex rollback 和 AvcsAgent 本地 rollback 都不撤销本地 `work/` 或 `output/` 文件；Avcs 只隐藏旧路径关联的当前消息和 Output board item。
5. 保存失败时前端保留编辑草稿，不退出编辑态。

## 6. WebSocket 事件

客户端发送：

1. `thread:items:list`：读取当前 thread 的聊天内容。
2. `message:send`：发送用户消息、当前聊天输入区引用的 asset 列表，以及可选 `data_provider`。
3. `message:edit_rerun`：编辑历史起始用户消息，并从该点重新运行 Agent。
4. `turn:stop`：停止当前 thread 的 active turn；后端经当前 Agent Harness 调用 interrupt 能力，queued turn 可在后端取消排队。AvcsAgent 中断当前 HTTP stream / Task 后把 turn 收敛为 `interrupted`。

服务端推送：

1. `agent:run_started`：Agent 开始运行。
2. `assistant:delta`：Agent 文本流式输出片段。
3. `item:created`：新增 turn/item。
4. `item:updated`：历史编辑、审批响应或工具状态原地更新。
5. `thread:items`：历史编辑重跑后刷新当前有效消息列表。
6. `tool:updated`：工具调用状态更新。
7. `agent:run_completed`：Agent 运行完成。
8. `error`：可恢复错误或失败状态。

## 7. 状态与错误

需要覆盖：

1. 当前 thread 为空。
2. Agent 正在运行。
3. Agent 调用失败。
4. 图片生成失败。
5. WebSocket 断开、重连中或重连失败。
6. 工具调用失败。
7. 用户主动停止 turn；状态展示为 stopped/interrupted，不按普通失败渲染。

错误应落在当前 turn 中，并保留用户输入，避免用户无法判断失败发生在 Agent、文件系统还是前端加载。

## 8. 验收标准

1. 用户可以在中栏发送消息，消息被保存到当前 thread。
2. 用户消息、Assistant 消息、工具调用、工具结果和错误都能按 item 展示。
3. Agent 输出支持流式或准流式展示。
4. 图片结果在聊天中以资产行展示，并可定位到画板对象。
5. Agent 运行中空 composer 的主按钮显示 Stop；非空 composer 可发送追加输入到当前 active turn，且用户仍可编辑草稿。
6. 用户可以通过调色盘为本轮 prompt 选择比例、数量和透明背景；发送成功后配置恢复默认。
7. 用户可以通过 Data Provider 入口选择 `avcs-data-prodiver-apod`，发送后后端持久化 provider 元信息，Agent runner 按 provider -> 图片生成的顺序执行。
8. 通过 `mix mock_many_turns` 生成的本地离线数据可被 `thread:items:list` 正常返回并展示到消息区。
9. Composer settings 摘要在当前 thread 未设置默认 Agent 参数时，使用全局软件设置作为 fallback。
10. 用户编辑已完成的起始用户消息后，后续旧路径消息从当前列表消失，系统从编辑后的消息重新运行 Agent。
